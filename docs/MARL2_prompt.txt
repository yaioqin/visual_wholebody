# 新增重点：base command 从 3D 扩展为 5D

请检查当前代码中的 `self.commands` 维度。

如果当前 command 还是 3D：

```python
[v_x, v_y, yaw_rate]
```

请新增配置开关：

```yaml
use_5d_base_command: true
```

当 `use_5d_base_command = true` 时，base command 应扩展为 5D：

```python
base_command = [
    v_x,
    v_y,
    yaw_rate,
    base_pitch,
    base_height
]
```

当 `use_5d_base_command = false` 时，保持旧的 3D command 流程不变，保证旧训练兼容。

# 1. Config 中新增 command range

请在 `b1z1_config.py` 或当前任务对应 config 中新增/修改：

```python
class commands:
    if use_5d_base_command:
        num_commands = 5
    else:
        num_commands = 3

    class ranges:
        lin_vel_x = [-0.5, 0.5]
        lin_vel_y = [-0.3, 0.3]
```

2. 不要把高度范围硬编码在 env 逻辑里，必须从 config 读取。
3. 如果当前机器人默认站立高度不是这个范围，请根据当前代码中的默认 base height / init state 调整，但必须保守。
4. 如果是 B1+Z1，建议先用：

```python
base_height = [0.32, 0.45]
```

5. 如果后续切到 B2+Z1，可以再改为：

```python

```

当前先按 B1Z1 默认使用 `[0.32, 0.45]`。

# 3. 使用 command 前必须 clip

在 command 采样后，或者每次使用 command 前，请添加安全 clip。

当 `use_5d_base_command = true` 时：

```python
self.commands[:, 3] = torch.clamp(
    self.commands[:, 3],
    self.cfg.commands.ranges.base_pitch[0],
    self.cfg.commands.ranges.base_pitch[1],
)

self.commands[:, 4] = torch.clamp(
    self.commands[:, 4],
    self.cfg.commands.ranges.base_height[0],
    self.cfg.commands.ranges.base_height[1],
)
```

目的：

1. 防止 command 超出物理可行范围。
2. 后续如果 high-level policy 输出 base command，也不会直接把机器人搞崩。
3. 训练时保证 base height command 始终合法。

# 5. Reward 中加入 base_height tracking

当前只训练 low-level policy，所以这里不是 high-level reward，而是底层 command tracking reward。

请在 reward 容器中新增可选 reward：

```python
reward_base_height_tracking
```

形式可以先用指数型 tracking：

```python
base_height_error = self.root_states[:, 2] - self.commands[:, 4]
reward_base_height = torch.exp(
    - torch.square(base_height_error) / self.cfg.rewards.tracking_height_sigma
)
```

请在 config 中新增：

```python
tracking_height_sigma = 0.05
```

以及 reward scale，例如：

```python
base_height_tracking = 1.0
```

如果当前 reward 系统使用类似 `scales` 的结构，请加入对应 scale。

注意：

1. 只有 `use_5d_base_command = true` 时才计算这个 reward。
2. 如果 `use_5d_base_command = false`，不要访问 `self.commands[:, 4]`，避免越界。
3. `base_height` tracking 的 scale 不要过大，避免压制 locomotion velocity tracking。
4. 如果当前代码已有 height tracking reward，请不要重复添加，而是改成跟踪 `self.commands[:, 4]`。

# 7. 与 arm-base message 的关系

当前训练目标仍然是：

```text
ActorCritic outputs 18D action
ManipLoco.step consumes 18D action

actions[:, :12]  -> leg joint target
actions[:, 12:18] -> one-step arm EE delta -> IK -> arm joint target
```

同时：

```text
H = 1
h = 1
gripper disabled
high-level disabled
```

`m_a2b` 仍然只在：

```yaml
use_arm_base_message: true
```

时加入 observation。

`m_a2b` 使用：

```python
self.last_arm_delta_action.view(num_envs, 1, 6)
```

和 Isaac Gym arm-only Jacobian：

```python
J_arm: [num_envs, 6, 6]
```

计算得到：

```python
m_a2b: [num_envs, 5]
```

注意：`base_height` command 和 `m_a2b` 是两个不同概念。

最终 observation 中可能同时包含：

```python
command_obs: [num_envs, 5]
m_a2b: [num_envs, 5]
```
